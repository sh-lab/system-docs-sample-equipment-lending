# エンティティ設計書（Entity Design）

## 1. エンティティID・名称
- エンティティID：`HFP-EL-E003_lending-request`
- 名称：`LendingRequest / 貸出申請`

---

## 2. 目的・スコープ
- 目的：
  - 本エンティティは、利用者による貸出申請から管理者の承認または却下、返却申請、返却確認までの業務進行を一貫して管理するための永続化単位である。
- スコープ：
  - 本エンティティが扱う情報
    - 申請者
    - 申請状態
    - 申請コメント、承認または却下コメント、返却申請コメント、返却確認コメント
    - 業務イベントごとの発生日時
    - 対象備品の明細一覧
  - 本エンティティでは扱わない情報
    - 備品マスタやユーザーマスタの表示用詳細情報
    - 画面表示順や検索専用の集計結果
    - 通知送信履歴や監査ログの詳細
- 参照：
  - 関連ユースケース：`UC-002`、`UC-003`、`UC-004`、`UC-005`、`UC-006`
  - 関連機能要件：`FR-002`、`FR-003`、`FR-004`、`FR-005`、`FR-006`、`FR-007`
  - 関連アプリケーションサービス：後続設計で定義する

---

## 3. 構造・境界
本エンティティは、貸出申請本体と、それに紐づく貸出申請明細で構成する。

- 本体エンティティとして扱う要素
  - 申請を一意に識別する情報
  - 申請者および処理担当者
  - 申請状態と業務コメント
  - 申請・承認・返却・完了に関する日時
- 本体エンティティに付随する要素
  - 貸出申請明細
  - 明細は対象備品を列挙するための従属要素であり、単独では存在しない
- 本エンティティとしての操作および管理の単位
  - 貸出申請1件を生成・更新・参照の単位とする
  - 明細の追加・変更は、貸出申請エンティティ全体の整合性のもとで行う

## 5. 不変条件（Invariants）

### 本体エンティティ
- 貸出申請ID は一意であり変更不可である。
- 貸出申請ID は登録時にデータベースシーケンスから採番される。
- 申請状態は `承認待ち`、`貸出中`、`返却確認待ち`、`却下`、`完了` のみを許可する。
- 申請者ユーザーは `利用者` ロールでなければならない。
- 承認または却下者、返却確認者が設定される場合、それぞれ `管理者` ロールでなければならない。
- `承認待ち` 状態では、承認または却下者ユーザーIDおよび返却確認者ユーザーIDを設定してはならない。
- `貸出中` または `却下` 状態では、承認または却下者ユーザーIDが設定されていなければならない。
- `返却確認待ち` 状態では、返却申請日時が設定されていなければならない。
- `完了` 状態では、完了日時が設定されていなければならない。

### 明細要素を含む条件
- 貸出申請明細は 1 件以上でなければならない。
- 同一貸出申請内で同じ備品を重複登録してはならない。

---

## 6. 状態モデル
- 許可される遷移：
  - `承認待ち`（`PENDING_APPROVAL`）→ `貸出中`（`LENT`）
    - 管理者が承認を確定したときに遷移する。
  - `承認待ち`（`PENDING_APPROVAL`）→ `却下`（`REJECTED`）
    - 管理者が却下を確定したときに遷移する。
  - `却下`（`REJECTED`）→ `完了`（`COMPLETED`）
    - 利用者が却下内容を確認したときに遷移する。
  - `貸出中`（`LENT`）→ `返却確認待ち`（`PENDING_RETURN_CONFIRMATION`）
    - 利用者が返却申請を行ったときに遷移する。
  - `返却確認待ち`（`PENDING_RETURN_CONFIRMATION`）→ `完了`（`COMPLETED`）
    - 管理者が返却確認を確定したときに遷移する。
- 禁止される遷移：
  - `承認待ち` から `完了`
  - `貸出中` から `却下`
  - `却下` から `貸出中`
  - `返却確認待ち` から `貸出中`
  - `完了` を起点とする遷移

## 9. 操作・設計上の制約
- 貸出申請本体および貸出申請明細の生成・更新・削除は、アプリケーションサービス経由でのみ行う。
- 貸出申請明細単体の Repository / Service は作成しない。
- 永続化は貸出申請エンティティ単位で行う。
- 備品状態の変更は、貸出申請状態の変更と整合していなければならない。
- 実装上の貸出申請エンティティは `src/main/java/net/shlab/hogefugapiyo/equipmentlending/model/entity/LendingRequest.java` に配置する。
- 実装クラスは `AuditVersionEntity` を継承する通常クラスとして定義し、`record` は用いない。
- 申請状態は `src/main/java/net/shlab/hogefugapiyo/equipmentlending/model/value/LendingRequestStatus.java` の enum で表現する。
- 明細の正データは `T_LENDING_REQUEST_DETAIL` として永続化し、履歴 record とは分離する。
- 貸出申請用 Entity Repository は Spring Data JPA および `EntityManager` を用いて実装し、
  本体と明細を同一エンティティ境界で扱う。
- 貸出申請ID の採番は、`T_LENDING_REQUEST` 用に定義したデータベースシーケンスを用いて行う。

---

## 10. 監査・バージョニング
- 監査情報はシステムが自動設定する。
- 併存更新は JPA の `@Version` による楽観ロックで制御する。
- 本体の状態変更が正常終了した場合の操作履歴は `H_LENDING_REQUEST_HISTORY` に記録する。
- 明細の登録履歴は `H_LENDING_REQUEST_DETAIL_HISTORY` に記録し、貸出申請エンティティの業務属性には含めない。
- 操作履歴は `operationId` 単位の追跡情報であり、エンティティ本体や明細の状態を復元するための正データとしては扱わない。
- 履歴 record は `src/main/java/net/shlab/hogefugapiyo/equipmentlending/model/history` 配下に配置する。
- 永続化は `EntityRepository` を継承した貸出申請用 Repository を通じて行う。
