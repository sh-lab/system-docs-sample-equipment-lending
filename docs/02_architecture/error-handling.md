
# エラー・例外処理方針

## 1. 基本方針
- 例外は原則として早期検知・早期通知を徹底する。
- 業務例外・システム例外ともに非検査例外（実行時例外）として定義する。
  - 検査例外はフレームワークのトランザクション制御との親和性が低く、呼び出し元に不要な catch を強制するため採用しない。
- 例外ハンドリングはグローバル例外ハンドラに集約し、個々の Controller での try-catch を原則として排除する。

## 2. 例外の分類
| 分類       | 説明                         | 例                       |
|------------|------------------------------|--------------------------|
| 業務例外   | 業務ルール違反（入力値不正は除く） | 在庫不足、重複申請、状態不整合等 |
| システム例外 | システム障害や外部要因等     | DB障害、通信エラー等     |

- 入力値不正（リクエストパラメータの未入力・形式不正等）は業務例外ではなく、コントローラ層でバリデーションエラーとして処理し、業務ロジック層には到達しない。

## 3. 例外ハンドリングの責務配置

### 3.1 グローバル例外ハンドラ（原則）
- グローバル例外ハンドラを設置し、例外種別に応じた共通処理を一元的に行う。
- 業務例外：エラーメッセージを解決し、遷移元画面へリダイレクトする。
- システム例外：エラーログを出力し、汎用エラー画面へ遷移する。

### 3.2 Controller 個別ハンドリング（例外的に許容）
- 以下の条件をすべて満たす場合に限り、Controller 内での個別 try-catch を許容する。
  1. エラー発生時に同一画面を再描画する必要がある。
  2. 再描画にあたり、リクエストパラメータ（フォーム入力値等）の保持が必要である。
  3. グローバル例外ハンドラではリクエストパラメータへのアクセスが困難である。
- 上記に該当しない場合は、グローバル例外ハンドラに委ねること。

### 3.3 業務ロジック層での方針
- 原則として、例外発生時は個々の業務ロジックで何もせず、上位層に委ねる。
- 局所的なリカバリやリトライが本当に必要な場合のみ、限定的に try-catch を使用する。

## 4. 例外発生時の処理
| 処理           | 業務例外                         | システム例外                   |
|---------------|----------------------------------|-------------------------------|
| ログ出力       | 必要に応じて WARN レベルで記録    | ERROR レベルでスタックトレース付きで記録 |
| ユーザー通知   | 業務メッセージを画面に表示        | 汎用エラー画面へ遷移           |
| トランザクション | ロールバック                     | ロールバック                   |

## 5. 例外クラス設計指針
- 業務例外：`BusinessException` 等の基底クラスを継承する。メッセージIDを保持し、表示メッセージの解決はハンドラ側で行う。
- システム例外：`SystemException` 等の基底クラスを継承する。原因例外（cause）をチェーンし、根本原因の追跡を可能にする。
- インフラ層で発生する技術例外（DB 障害、シーケンス払い出し失敗等）は `SystemException` でラップし、上位層に伝播させる。
- 命名規則：`XXXException`

## 6. その他
- 外部API連携時の例外は、API種別ごとに個別ハンドリング方針を定める。
