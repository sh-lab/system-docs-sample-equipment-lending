# テスト方針

## 1. 目的

本書は、本リポジトリにおけるテストの目的、範囲、実施方針、および品質判断の基準を定義する。

本書では、個別テストケースの詳細ではなく、実装および変更時に共通して参照するテスト方針を定義する。

---

## 2. 適用範囲

- 本方針は、本システムに対する新規実装、改修、不具合修正に適用する。
- 自動テスト、手動確認、レビューによる確認を含め、品質確保のための確認活動全体を対象とする。
- `docs/02_architecture/`、`docs/03_designs/`、`docs/04_implementation/` に記載された方針と整合することを前提とする。

---

## 3. テストの基本方針

- テストは、仕様どおりに動作することの確認だけでなく、変更による意図しない影響の検出を目的とする。
- テストの実施方法は、対象の責務、リスク、変更影響範囲に応じて選択する。
- テストは後工程でまとめて行うのではなく、実装とあわせて継続的に実施する。
- 自動化可能な確認は、再現性と継続性を考慮して優先的に自動化を検討する。
- 文書のみの変更であっても、関連文書間の整合、文体、参照関係は確認対象とする。

---

## 4. テストレベルと観点

### 4.1 テストレベル

- 単体レベル
  - Pure Service、Application Service、Command Service を中心に、個別の判定、変換、計算、状態遷移を確認する。
- 結合レベル
  - Web スライステストおよび Repository テストにより、複数要素の連携を確認する。
- 業務フロー / 受け入れレベル
  - 限定した Integration Test により、代表的な業務動線が達成できることを確認する。

### 4.2 主な確認観点

- 正常系
  - 期待する入力に対して正しい結果を返すこと。
- 異常系
  - 不正入力、前提条件違反、業務制約違反時に適切に失敗すること。
- 境界値
  - 最小値、最大値、未設定、空、件数上限などの境界条件で破綻しないこと。
- 回帰
  - 既存仕様への影響がないこと。
- 整合性
  - データ、状態、メッセージ、画面表示、外部連携結果が矛盾しないこと。

---

## 5. 本リポジトリにおけるテスト種別ごとの方針

### 5.1 単体テスト

- 対象は Pure Service、Application Service、Command Service とする。
- 目的は、分岐、境界条件、例外、業務ルールが仕様どおりであることの保証である。
- 1クラス = 1テストクラスを原則とする。
- 1テストメソッド = 1シナリオを原則とする。
- テストは時間、乱数、外部 I/O に依存させず、決定性を保つ。
- 時刻依存がある場合は、業務コードでは `CurrentTimeProvider` を利用し、実行時刻によって結果がぶれない構造を優先する。
- テストでは `CurrentTimeProvider` の差し替え、または `TimeConfiguration` で定義した `Clock` Bean の固定により、実行時刻によって結果が変わらないようにする。
- 基本技術要素は JUnit 5 とし、Mockito の利用は必要最小限に留める。

### 5.2 Web スライステスト

- 対象は Spring MVC Controller とする。
- 目的は、ルーティング、バリデーション、PRG、重複送信防止、View 名、および Controller が薄い責務に留まっていることの確認である。
- Controller の業務ロジックは直接検証せず、Application Service をモック化して Controller 責務に集中する。
- 基本技術要素は `@WebMvcTest`、MockMvc、`@MockitoBean` とする。
- 主な確認観点は以下とする。
  - GET 初期表示が正しい View を返すこと。
  - POST 成功時は redirect となり、PRG に従うこと。
  - バリデーションエラー時は同画面を再表示すること。
  - メッセージ種別（info / warning など）が正しく分岐すること。
  - 管理対象メッセージを表示する場合、`[メッセージID]メッセージ本文` 形式になっていること。
  - Spring Security を利用する更新系リクエストでは、CSRF トークン付与を前提に確認すること。
- 重複送信防止対象フォームでは、`HandlerInterceptor` によるワンタイムトークン正常系と、欠落または再利用時の専用エラー画面へのリダイレクト異常系を確認すること。

### 5.3 Repository テスト

- 対象は Entity Repository および Query Repository とする。
- 目的は、クエリ条件、0件時の挙動、並び順、件数上限の適用を保証することである。
- 1 Repository = 1テストクラスを原則とする。
- 全条件の網羅ではなく、代表的なクエリを対象に確認する。
- 基本技術要素は `@DataJpaTest` と H2 とする。
- テスト後はロールバックされる前提で、メソッドごとの独立性を保つ。
- 主な確認観点は以下とする。
  - 条件一致時と不一致時（0件）の結果が正しいこと。
  - 件数上限が正しく適用されること。
  - 想定外条件でも異常に暴れないこと。

### 5.4 限定 Integration Test

- 対象は主要な業務ユースケースのうち、代表的な 1〜2 本とする。
- 目的は、画面、Application Service、Repository の接続が正しく疎通していることの確認である。
- Integration Test は乱用せず、代表動線の保証に限定する。
- 基本技術要素は `@SpringBootTest` とし、実 DB は用いず H2 を使用する。

### 5.5 文書変更の確認

- 文書のみの変更では、自動テストの追加実行を必須としない。
- ただし、対象文書とテンプレート、関連設計書、既存実装規約との整合をレビューまたは差分確認で検証する。

---

## 6. 実施タイミング

- 実装着手前
  - 対象変更のリスクと必要な確認観点を整理する。
- 実装中
  - 小さな単位で検証を行い、問題を早期に検出する。
- 実装完了時
  - 変更対象の責務に応じた確認を一通り実施する。
- レビュー前後
  - レビュー指摘や設計変更に伴う影響範囲を再確認する。

---

## 7. テストデータ・環境の方針

- テストに使用するデータは、確認したい条件が明確に表現できるものを用意する。
- 正常系と異常系で必要な前提データを分けて管理し、再現可能な状態を保つ。
- テストの再実行により結果が不安定にならないよう、前提状態を可能な限り制御する。
- 実装テストの代表的な実行コマンドは `./mvnw test` とする。
- データアクセス系および結合系の確認では、実 DB ではなく H2 を前提とする。
- 文言確認では、句読点、件数、状態名称などの差異も確認対象に含める。

---

## 8. 自動化の方針

- 頻繁に実施する確認、回帰確認に有効な確認、人的判断を必要としない確認は自動化を優先する。
- 自動テストは、失敗時に原因の切り分けがしやすい粒度で構成する。
- 自動化しない確認については、手動で確認する理由と観点を明確にする。
- 自動化の有無にかかわらず、変更内容に応じて必要な確認を省略しない。
- Controller、画面遷移、メッセージ表示、検索条件、一覧表示などは、既存の Spring Boot テストおよび MockMvc を活用して回帰確認する。

---

## 9. レビュー観点

- テスト観点が対象の責務と変更内容に対応しているか。
- 正常系だけでなく、異常系、境界値、回帰が考慮されているか。
- 確認方法が再現可能であり、実施者によって結果が大きくぶれないか。
- 文書変更の場合、関連文書との整合およびテンプレートとの差分意図が説明可能か。

---

## 10. 結果判定と完了条件

- 期待した結果と実際の結果を比較し、差異がある場合は原因を明確にする。
- 業務上重要な不整合や再現性のある不具合が残っている状態では完了としない。
- 軽微な既知課題を残す場合は、理由、影響範囲、対応方針を明示する。
- Java 実装変更時は、影響範囲に応じた自動テストを実行したうえで完了とする。
- 画面変更時は、表示文言、ラベルキー、メッセージ種別、遷移先、ボタンの有効 / 無効状態まで確認して完了とする。

---

## 11. 変更管理

- 本方針に影響する変更が生じた場合は、関連する設計方針、実装規約、運用ルールとの整合を確認する。
- テスト方針の大きな変更は、理由と判断背景が追跡できるように記録する。
