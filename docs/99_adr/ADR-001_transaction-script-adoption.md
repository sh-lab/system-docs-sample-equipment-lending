# ADR-001 トランザクションスクリプト方式の採用

## ステータス
採用

## コンテキスト
本サンプルアプリケーションは、
業務システム開発における設計から実装への落とし込みを
学習することを目的としている。

本プロジェクトでは、ADR-000 にて
system-docs-template を前提とした設計ドキュメント構成を採用している。

その上で、アーキテクチャ方式についても学習用途に適した前提を明文化する必要がある。

特に本サンプルでは、
業務処理の流れを設計書のユースケースや画面仕様と対応付けて
理解しやすくすることを重視している。

一方で、リッチドメインモデルを採用すると、
サンプルとしては責務の見通しが複雑になりやすい。

そのため、実装方式は以下の観点を満たす必要がある。

- ユースケースごとの処理の流れを追跡しやすいこと
- 設計書と実装コードの対応関係を説明しやすいこと
- 学習者にとって責務分担が理解しやすいこと
- GitHub Copilot を用いた実装補助でも一貫性を保ちやすいこと

## 決定
本サンプルアプリケーションでは、
業務処理の実装方式として
**トランザクションスクリプト方式**を採用する。

ユースケース単位の処理フローは Application Service を中心に構成し、
書き込み処理は Command Service、
読み取り処理は Query Service、
副作用を持たない判定・計算は Pure Service に分離する。

また、Entity はデータ保持と状態表現に専念し、
業務ロジックを持たせない。

## 理由
- 学習用途として、処理の流れが
  Application Service から順に読める構造の方が理解しやすいため。
- ユースケース単位の設計と
  `1ユースケース = 1メソッド` の実装規約を対応付けやすいため。
- トランザクション境界を Application Service に置くことで、
  更新処理の責務と単位を明確にできるため。
- Entity をデータ保持に限定することで、
  設計書の Entity 定義と実装上の役割を過度に乖離させずに済むため。
- GitHub Copilot に対しても、
  レイヤごとの責務と実装パターンを明示しやすく、
  出力のばらつきを抑えやすいため。

## 代替案
- リッチドメインモデル方式を採用し、
  Entity に業務ロジックや整合性制御を集約する。
- Domain Service を中心とした設計を採用し、
  業務ルールをより細かい抽象化で表現する。
- Controller から直接 Repository や永続化処理を呼び出す
  単純な CRUD 中心構成とする。

これらはいずれも一定の妥当性はあるが、
本サンプルでは以下の理由により採用しない。

- リッチドメインモデル方式は、
  本サンプルの業務複雑度に対してオーバーエンジニアリングとなり、
  学習の焦点がドメインモデリング技法に偏るため。
- Domain Service 中心の設計は、
  サービスとエンティティの責務境界が曖昧になりやすく、
  学習初期段階では混乱を招くため。
- CRUD 中心構成は、
  層の責務分担が学べず、実務への応用性が低いため。

## 結果・影響
- Application Service は、
  ユースケース単位で処理順序とトランザクション境界を定義する。
- Command Service は状態変更と永続化を担当し、
  Query Service は参照専用の取得処理を担当する。
- Pure Service は副作用を持たない判定・計算に限定する。
- Entity はデータ構造および状態保持に専念し、
  リッチ Entity として振る舞わせない。
- 実装規約および Copilot への指示は、
  本 ADR を前提として `docs/04_implementation` 配下に整理する。
- 本方式は業務ルールが比較的単純な本サンプルに適した選択である。
  ドメインルールの複雑化（エンティティ間の整合性制約や
  状態遷移ルールの増大等）が見込まれる場合は、
  エンティティ自身に業務ロジックを持たせる
  リッチドメインモデル方式への移行を検討する。
