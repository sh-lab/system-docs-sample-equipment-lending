# 設計から実装への落とし込み

本ドキュメントでは、本リポジトリにおいて  
**設計書がどのように実装へ落とし込まれているか**を説明する。

ここで扱うのは、実装技法そのものではなく、

- 設計書とコードの対応関係
- 設計単位が実装単位にどう写像されているか
- 変更時にどこを見ればよいか

といった、**設計と実装の接続点**である。

---

## 本リポジトリにおける前提

本リポジトリでは、次の前提を置いている。

- 実装は必ず設計書に基づいて作成されている
- 実装を理解するためには、対応する設計書を参照する
- 設計書は実装の写しではなく、実装の前提条件である

そのため、「コードから読む」のではなく、  
**設計書 → コード** の順で追うことを推奨する。

---

## 設計書の構造（再確認）

`docs/03_designs/` には、system-docs-template を前提とした  
設計書が配置されている。

```txt
docs/03_designs/ 
├─ data 
├─ entity 
├─ services 
│  ├─ application 
│  ├─ command 
│  ├─ query 
│  └─ pure 
└─ ui
```

この構成は、  
**どの種類の設計が、どの種類のコードに対応するか**  
を明確にするためのものである。

---

## UI設計とController

### UI設計の役割

`docs/03_designs/ui/` では、

- 画面構成
- 項目
- 画面遷移
- 入力制約

といった、**ユーザーとの接点**を定義している。

UI設計は、「Controller が何を受け取り、何を返すか」の前提となる。

---

### 実装への対応

- UI設計  
  → Controller のエンドポイント、DTO、Validation

Controller には、

- 業務判断
- 状態遷移
- データ取得ロジック

は書かれない。

Controller はあくまで、  
**UI と Application Service を接続する薄い層**として実装されている。

---

## Application Service設計と実装

### 設計上の役割

`docs/03_designs/services/application/` では、

- ユースケース単位の処理の流れ
- Command / Query の組み合わせ
- トランザクション境界

を定義している。

ここでの設計書は、  
「1ユースケース = 1 Application Service」を基本単位としている。

---

### 実装への対応

- Application Service 設計書  
  → Application Service クラス

実装では、

- フロー制御
- 例外変換
- レスポンス組み立て

を担当し、  
業務ルールそのものは持たない。

---

## Command / Query Service設計と実装

### Command Service

`docs/03_designs/services/command/` では、

- 状態遷移
- 業務ルール
- 整合性チェック

を定義している。

設計書1枚が、  
基本的に **1つの Command Service** に対応する。

実装では、

- 永続化
- 履歴記録
- Pure Service の呼び出し

などが集中する。

---

### Query Service

`docs/03_designs/services/query/` では、

- 検索条件
- 取得項目
- 集計単位

を定義している。

設計書1枚が、  
**1つの Query Service** に対応する。

Query Service は、

- 状態を変更しない
- 副作用を持たない

という制約のもとで実装されている。

---

## Pure Service設計と実装

`docs/03_designs/services/pure/` では、

- 計算
- 判定
- 再利用可能な業務知識

を定義している。

Pure Service は、

- 単体でテストできる
- 複数の Command / Query から呼ばれる

ことを前提に、  
**ステートレスなクラス**として実装されている。

---

## Entity設計と実装

### Entity設計

`docs/03_designs/entity/` では、

- 業務としてのエンティティ
- 状態
- 振る舞い

を定義している。

Entity 設計は、  
**データ構造ではなく、業務概念の定義**である。

---

### 実装への対応

- Entity設計  
  → JPA Entity クラス

ただし、本教材はトランザクションスクリプト方式を採用しているため（`reading/03` 参照）、  
Entity にすべての業務ロジックを集約することは意図していない。

状態遷移や整合性の制御は、  
Command Service / Pure Service に配置する方針のもとで実装されている。

---

## データ設計と実装

### データ設計

`docs/03_designs/data/` には、

- テーブル定義
- 履歴テーブル
- リレーション

が整理されている。

これは、

- Repository 実装
- SQL / クエリ

の前提情報として利用される。

---

### 実装への対応

- データ設計  
  → Entity Repository / Query Repository

履歴系テーブルは、  
**History Repository** からのみ操作されることが前提となっている。

---

## 設計と実装を対応づけて読むための視点

設計書とコードを対応づけて読む際には、次を意識するとよい。

- 設計書のIDとクラス名が対応しているか
- 設計書が定義している責務が、実装で破られていないか
- 設計書に書いていない判断が、実装に入り込んでいないか

「動いているから正しい」ではなく、

> **設計書が意図していた構造になっているか**

を確認することが重要である。

---

## 実装規約（docs/04_implementation）について

本リポジトリでは、設計書とは別に  
`docs/04_implementation/` 配下に実装規約を配置している。

これは、設計で定義した責務分担や構造を
実装として破綻させないための判断基準をまとめたものであり、
設計そのものを定義する文書ではない。

実装方針や具体的な制約については、
`docs/04_implementation/implementation-rules.md`
を参照されたい。

---

## 演習課題との関係

演習課題では、

- 要件変更
- 設計修正
- 実装反映

を順に行う。

実装を修正する前に、

1. どの設計書が影響を受けるか
2. Application / Command / Query のどこに変更が入るか
3. Pure Service として切り出すべきか

を整理することが、  
設計を起点とした開発プロセスの練習となる。

---

## まとめ

- 設計書は、実装の補足説明ではない
- 実装は、設計書の結果として存在する
- 設計と実装の対応関係を追えることが、保守性につながる

本リポジトリでは、  
**「どの設計を見れば、どのコードに影響があるか」**  
を把握できる構造を最重要視している。
