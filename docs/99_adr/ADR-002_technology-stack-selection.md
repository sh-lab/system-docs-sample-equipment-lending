# ADR-002 技術スタックの選定（サンプルアプリ）

## ステータス
採用

## コンテキスト
本サンプルアプリケーションは、
業務システム開発における設計から実装への落とし込みを
学習することを目的としている。

本プロジェクトでは、ADR-000 にて
system-docs-template を前提とした設計ドキュメント構成を採用しており、
実装についても「実務で一般的に想定される構成」を
過度な抽象化や最適化を行わずに示すことを重視する。

そのため、技術スタックは以下の観点を満たす必要がある。

- 業務システムでの採用事例が多いこと
- 学習者にとって情報が入手しやすいこと
- 設計と実装の対応関係を説明しやすいこと
- 環境構築および実行の再現性が高いこと

## 決定
本サンプルアプリケーションでは、以下の技術スタックを採用する。

- プログラミング言語：**Java 21**
- アプリケーションフレームワーク：**Spring Boot**
- テンプレートエンジン：**Thymeleaf**
- 永続化方式：**JPA**
- データベース：**H2（インメモリ）**
- ビルドツール：**Maven**

## 理由

### Java 21 の採用
- 長期サポート（LTS）版であり、言語仕様として安定しているため。
- 業務システム開発における採用事例が多く、
  学習者が既存知識と結び付けやすいため。
- system-docs-template に基づく設計
  （Entity / Application Service / Repository）
  を素直に実装へ落とし込みやすいため。

### Spring Boot の採用
- 業務アプリケーション開発での採用事例が非常に多く、
  学習用途として妥当であるため。
- 設定の記述量が比較的少なく、
  アーキテクチャおよび設計意図の説明に集中できるため。
- Controller / Application Service / Infrastructure といった
  レイヤ構造を明示しやすいため。
- `spring-boot-starter-validation` を用いることで、
  Controller の基本バリデーションを Spring Boot の標準機能へ統一できるため。
- メッセージプロパティと組み合わせることで、
  入力不備を画面へ一貫した形で返す実装を学習しやすいため。

### Thymeleaf の採用
- Spring Boot と親和性が高く、
  サーバサイドレンダリングによる画面実装例を示しやすいため。
- HTML テンプレートとしてそのまま閲覧可能であり、
  画面設計書と実装結果の対応関係を理解しやすいため。
- 画面表示ロジックと業務ロジックの責務分離
  （Controller / Application Service）を説明しやすいため。
- 業務システムにおける従来型の MVC 構成を
  学習用途として示すのに適しているため。

### JPA の採用
- Java / Spring 系業務システムにおける採用事例が多く、
  学習用途として一般性が高いため。
- Repository を通じた永続化の責務分離を、
  実装例として示しやすいため。
- Entity とテーブルの対応関係を整理しやすく、
  設計ドキュメントと実装のつながりを理解しやすいため。
- Spring Boot と組み合わせた構成が一般的であり、
  GitHub Copilot による補助でも一貫したコード生成を期待しやすいため。
- Entity Repository の実装では Spring Data JPA および `EntityManager` を利用し、
  `@Entity` と `@Version` を用いた標準的な永続化および楽観ロックを
  学習対象として明示しやすいため。
- 参照系の Query Repository および履歴登録は、検索最適化や単純書き込みの責務を持つため、
  本サンプルでは JDBC ベース実装とし、Entity Repository の JPA 実装と役割を分離しやすいため。

### H2（インメモリ）の採用
- サンプル用途として、環境構築のコストが低く、
  再現性が高いため。
- 起動および破棄が容易であり、
  学習者が手元で試行錯誤しやすいため。
- 本サンプルは本番運用を目的としないため、
  性能や運用面での制約は評価対象外とする。

### Maven の採用
- Java 業界において広く利用されており、
  業務システムのサンプルとして一般性が高いため。
- 設定が宣言的であり、
  依存関係およびビルドの流れを追いやすいため。
- 学習用途として、設計と実装の対応関係に
  集中しやすい構成であるため。
- GitHub Copilot によるコード補助において、
  定型的な `pom.xml` 構成が安定して生成されやすいため。

## 代替案
以下の代替技術も検討対象となり得るが、本サンプルでは採用しない。

- Kotlin + Spring Boot
- フロントエンド SPA（React / Vue 等）を採用する構成
- JDBC 直書きや MyBatis 等を採用する構成
- PostgreSQL / MySQL 等の外部データベース
- マルチモジュール構成
- Gradle をビルドツールとして採用する構成

これらは学習用途として複雑性が増し、
設計ドキュメントと実装の対応関係が
把握しにくくなる可能性があるため、
本サンプルの目的には適さないと判断した。

## 結果・影響
- 本サンプルの実装は、特定の業務システムを
  そのまま再現することを目的としない。
- 本番システムへの直接的な流用は想定しない。
- 画面設計書と Thymeleaf テンプレートの対応関係を通じて、
  設計から実装への流れを理解することを主目的とする。
- 永続化処理については、Entity Repository を Spring Data JPA / `EntityManager`
  ベースで実装し、Entity とテーブルの対応、更新、楽観ロックの流れを
  学習可能な構成として整理する。
- 入力チェックは Spring Boot Validation とメッセージプロパティを組み合わせ、
  Controller が UI 層の責務として基本バリデーションを担う構成とする。
- 参照系の Query Repository および履歴系 Repository は、
  本体エンティティの更新責務を持たないため、JDBC ベース実装とする。
- 実装補助情報および Copilot への指示は、
  `docs/04_implementation` 配下に整理して記載する。
