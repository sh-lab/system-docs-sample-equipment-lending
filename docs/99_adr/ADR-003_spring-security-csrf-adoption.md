# ADR-003 Spring Security と CSRF 対策の前提化

## ステータス
採用

## コンテキスト
本サンプルアプリケーションは、
業務システム開発における設計から実装への落とし込みを
学習することを目的としている。

本プロジェクトでは、ADR-002 にて
Spring Boot を中心とした技術スタックを採用しているが、
Web アプリケーションとして最低限どのセキュリティ前提を持つかは
明文化していなかった。

一方で、業務アプリケーションでは、
画面からの POST や更新系リクエストに対して
CSRF 対策を前提とすることが一般的である。

ただし、本サンプルの主題は
認証・認可の高度な設計やセキュリティ実装そのものではなく、
業務設計と実装の対応関係を学ぶことにある。

そのため、セキュリティについては
業務アプリの前提として最低限必要な範囲を採用しつつ、
本題ではない詳細設計とは切り分ける必要がある。

## 決定
本サンプルアプリケーションでは、
業務アプリケーションの前提を示す目的で
**Spring Security を導入し、CSRF 対策を有効とする。**

認証方式は学習用サンプルとして簡易的なものに留め、
業務ロジックとは分離して扱う。

認証・認可の詳細仕様、
権限制御モデル、
運用時のセキュリティ設計の深掘りは
本サンプルの主対象外とする。

## 理由
- 業務 Web アプリケーションにおける
  更新系リクエストの基本前提として、
  CSRF 対策を省略しない構成を示す必要があるため。
- 学習用サンプルであっても、
  実務で一般的なセキュリティ前提を外した構成は
  誤解を招きやすいため。
- Spring Security を用いることで、
  Spring Boot ベースの一般的な実装構成に沿って
  セキュリティ前提を示しやすいため。
- 一方で、認証・認可の詳細まで主題に含めると
  本サンプルが重視する業務設計と実装の対応関係が
  見えにくくなるため、
  範囲を限定する必要があるため。

## 代替案
- Spring Security を導入せず、
  セキュリティ考慮を持たない簡易 Web アプリとして構成する。
- CSRF を無効化し、
  画面・Controller・テストの簡潔さを優先する。
- 認証・認可まで含めて
  本格的なセキュリティ設計を本サンプルの主要テーマに含める。

これらはいずれも一部には合理性があるが、
本サンプルでは
「業務アプリとして最低限持つ前提」と
「学習対象として深入りしない範囲」の両立を優先するため採用しない。

## 結果・影響
- 画面からの POST 等の更新系リクエストでは、
  CSRF トークンの存在を前提として扱う。
- Web スライステストおよび画面テストでは、
  必要に応じて CSRF トークン付与を考慮する。
- 認証・認可の詳細設計は、
  本サンプルの説明範囲では簡易化されたものとなる。
- セキュリティ機能は業務ロジックと分離して扱い、
  学習対象の中心は引き続き業務設計と実装の対応関係とする。

### 認証関連サービスの設計書上の扱い

認証関連の実装（`LoginApplicationService`、`FindLoginUserQueryService`、`LoginController`）は、
業務ユースケースを実現するサービスではなく、認証基盤の一部として位置づける。

そのため、業務サービス設計書（SAS / SQS）の作成対象外とし、
認証に関する設計は `docs/03_designs/security-design.md` および
画面設計書 `docs/03_designs/ui/login.md` に集約する。

### 本サンプルにおける認証の簡易化

本サンプルでは、認証の実装を以下のとおり簡易化している。

- 全利用者に共通のパスワード（`pass`）を使用し、パスワードのハッシュ化を行わない。
- `M_USER` テーブルにパスワードカラムを持たない。
- Spring Security の `AuthenticationProvider` / `UserDetailsService` を使用せず、
  Controller 内で手動認証を行う。
- ログイン画面に試験用利用者一覧および共通パスワードを表示する。

これらは学習用途における環境構築コストの低減と、
認証処理への学習上の注意分散を避けることを目的とした割り切りである。

**本サンプルの認証構成を本番環境で使用してはならない。**
本番環境では、パスワードのハッシュ化保存、
ユーザーテーブルへの認証情報格納、
`AuthenticationProvider` 等を用いた標準的な認証方式を採用すること。

簡易化の設計上の詳細は `docs/03_designs/security-design.md` §4.4 を参照。
