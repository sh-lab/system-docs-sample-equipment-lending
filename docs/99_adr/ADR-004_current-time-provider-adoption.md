# ADR-004 CurrentTimeProvider による時刻取得の統一

## ステータス
採用

## コンテキスト
本サンプルアプリケーションは、
業務システム開発における設計から実装への落とし込みを
学習することを目的としている。

業務システムでは、
貸出期限判定、検索条件の日付比較、更新日時設定など、
現在時刻に依存する処理が自然に増えていく。

その際、業務コードの各所で
`LocalDate.now()`、`LocalDateTime.now()`、`Instant.now()`、
`System.currentTimeMillis()` などを直接呼び出すと、
以下の問題が起きやすい。

- テスト時に時刻を固定しにくい
- 同一ユースケース内で取得時刻が微妙にずれる
- 時刻取得の方針がクラスごとにばらつく
- 学習者が「どこで時刻を取得するのがよいか」を判断しにくい

一方で、すべての業務コードが `Clock` を直接扱う構成にすると、
Java の時刻 API と設計意図の両方を理解する必要があり、
学習用サンプルとしては判断軸が分散しやすい。

## 決定
本サンプルアプリケーションでは、
業務コードにおける現在時刻の取得を
`net.shlab.hogefugapiyo.framework.core.time.CurrentTimeProvider`
へ統一する。

`CurrentTimeProvider` の内部実装は
framework 共通部で `Clock` を利用し、
業務コードは `Clock` を直接扱わない。

Pure Service では `CurrentTimeProvider` を直接注入せず、
呼び出し元から時刻値または判定に必要な値を引数で受け取る。

## 理由
- 時刻依存の業務処理をテストしやすくし、
  実行時刻によるテストのぶれを防ぎやすくするため。
- 同一ユースケースで利用する現在時刻の取得責務を明確にし、
  実装の散在を防ぐため。
- `Clock` の利用箇所を framework 共通部へ集約し、
  業務コード側の判断を単純化するため。
- 学習者に対して、
  「業務コードでは `CurrentTimeProvider` を使う」
  という分かりやすい基準を提示するため。
- 時刻取得の抽象化を共通化することで、
  将来のタイムゾーン方針やテスト戦略の見直しにも対応しやすくするため。

## 代替案
- 業務コードの各所で `LocalDate.now()` や `LocalDateTime.now()` を直接呼び出す。
- 各クラスで `Clock` を直接注入して利用する。
- 時刻が必要な都度、独自の utility や static メソッドで取得する。

これらは実装自体は可能であるが、
テスト容易性、責務の一貫性、学習時の判断基準の明確さを損ないやすいため、
本サンプルでは採用しない。

## 結果・影響
- 業務コードでは、
  現在時刻が必要な場合に `CurrentTimeProvider` を利用する前提となる。
- framework 共通部では、
  `TimeConfiguration` により `Clock` Bean を定義し、
  `ClockCurrentTimeProvider` がそれを利用する構成となる。
- テストでは、
  `CurrentTimeProvider` の差し替え、
  または `Clock` Bean の固定により時刻依存を制御する。
- Pure Service は時刻取得手段そのものを持たず、
  呼び出し元から渡された値で判定する設計を優先する。
- 実装規約およびテスト方針では、
  時刻取得の標準手段として `CurrentTimeProvider` を参照する。
