# ID・命名体系（ID & Naming Conventions）

本書は、本システムにおけるIDおよび命名規則を横断的に定義し、
設計・実装・ログ・運用・問い合わせ対応において、
識別子および名称の一貫性を保つことを目的とする。

画面、サービス、エンティティ、テーブル、監査カラム、ログ出力など、
複数の設計ドキュメントにまたがる識別子・命名の基準は、
本書を正として統一する。

---

## 1. 利用目的
- 設計書・仕様書・テスト・運用・ログ・問い合わせ対応において、
  対象要素を一意に識別可能とする。
- コード・テンプレート・ドキュメント・ログにおける表記ゆれを防止し、
  管理性および追跡性を向上させる。

---

## 2. 共通命名ルール
- IDおよび物理名の先頭は必ず `HFP-EL-` とする。
  - `HFP` … 会社略称（HOGE_FUGA_PIYO）
  - `EL` … システム略称（EquipmentLending）

---

## 3. 画面IDとサービスIDの対応方針

### 3.1 基本方針
- 画面ID（View/UI）は、`100, 200, 300...` のように **100番刻み**で付与する。
- 画面に対応するサービスは、原則として **3桁連番の百の位（1桁目）を画面IDと一致**させることで関連付ける。
- 他アプリケーションサービスからの呼び出しを想定するサービスは、画面対応よりも横断利用時の識別性を優先し、`0xx` 採番を許容する。
- **1つのサービスIDは1つの処理を表す**。
- **1画面に複数のアプリケーションサービスが紐づく**ことを前提とする。

---

### 3.2 サービスIDの採番ルール
- 百の位（1桁目）：
  - 原則として対応する画面IDと一致させる。  
  - 例：`V100` → `SAS1nn / SQS1nn / SCS1nn / SPS1nn`
  - 例外として、他アプリケーションサービスからの呼び出しを想定するサービスは `0` を用いる。
- 下2桁（`nn`）：
  - 処理単位で `01, 02, 03...` のように採番する
- サービス種別ごとに独立した連番とする。
- `0xx` 採番を用いた場合でも、本文参照記法および命名規則は他サービスと同一とする。

---

### 3.3 設計書本文での参照記法
- 設計書内で他画面を参照する場合は、**`名称(ID)`** の形式で記載する。
- 画面参照で記載するIDは、`V100`、`V300` のように、**画面種別記号 + 3桁連番まで** とする。
- 設計書内でサービスを参照する場合も、**`名称(ID)`** の形式で記載する。
- サービス参照で記載するIDは、`SAS001`、`SQS001`、`SCS002`、`SPS001` のように、**サービス種別記号 + 3桁連番まで** とする。
- 参照記法では、`HFP-EL-` の共通接頭辞および後続の任意文字列は省略し、本文の可読性を優先する。

例：
- 画面：`備品検索画面(V300)`
- 画面：`利用者貸出申請・返却画面(V400)`
- サービス：`備品検索初期表示サービス(SAS301)`
- サービス：`備品検索情報取得サービス(SQS301)`

---

### 3.4 例：画面IDとの対応関係

#### 画面ID
- `HFP-EL-V100_user-mypage`（利用者マイページ）

#### 対応するサービス群
- アプリケーションサービス  
  - `HFP-EL-SAS101_user-mypage-init_service`
  - `HFP-EL-SAS102_start-equipment-search_service`
- クエリサービス  
  - `HFP-EL-SQS121_find-equipment-summary_service`
- コマンドサービス  
  - `HFP-EL-SCS111_apply-loan_service`
- ピュアサービス  
  - `HFP-EL-SPS131_check-lending-policy_service`

---

### 3.5 画面非対応サービス・外部利用想定サービス
- 特定の画面に直接対応しない汎用サービスは、百の位を `0` とする。
- 他アプリケーションサービスからの呼び出しを想定し、画面対応に閉じない識別子が必要なサービスも、百の位を `0` とする。
- 例：
  - `HFP-EL-SAS001_initialize-master-data_service`
  - `HFP-EL-SQS012_load-common-config_service`
  - `HFP-EL-SCS001_register-lending-request_service`
  - `HFP-EL-SPS001_check-lending-request-availability_service`

---

## 4. サービスID体系

### 4.1 アプリケーションサービス（SAS）
- フォーマット：`HFP-EL-SASXXX_[任意文字列]_service`
- ユースケースと1対1に対応し、業務処理およびトランザクション境界を表す。

### 4.2 コマンドサービス（SCS）
- フォーマット：`HFP-EL-SCSXXX_[任意文字列]_service`
- 状態変更処理を担う内部実装単位とする。

### 4.3 クエリサービス（SQS）
- フォーマット：`HFP-EL-SQSXXX_[任意文字列]_service`
- 参照処理を担う内部実装単位とする。

クエリサービス用リポジトリが必要な場合は、以下の形式を用いる。
- `HFP-EL-SQSXXX_[任意文字列]_repository`

### 4.4 ピュアサービス（SPS）
- フォーマット：`HFP-EL-SPSXXX_[任意文字列]_service`
- 副作用を持たない業務ロジックの実装単位とする。

### 4.5 サービス実装クラス内包 DTO の命名
- コマンドサービス実装クラス内に定義する入出力レコードは、以下の名称を原則とする。
  - リクエスト：`Request`
  - 履歴登録用レスポンス：`HistoryResponse`
- クエリサービス実装クラス内に定義する入出力レコードは、以下の名称を原則とする。
  - リクエスト：`Request`
  - レスポンス：`Response`
- 上記レコードは、原則として `public static record` とし、利用側は `契約インターフェース名.レコード名` の形式で参照する。
- DTO の配置、可視性、利用側参照の詳細は `../04_implementation/implementation-rules.md` を参照する。

---

## 5. エンティティID
- フォーマット：`HFP-EL-EXXX_[任意文字列]`
- 例：
  - `HFP-EL-E001_equipment`
  - `HFP-EL-E002_user`
  - `HFP-EL-E003_lending-request`

エンティティに対応するリポジトリは以下の形式とする。
- `HFP-EL-EXXX_[任意文字列]_repository`

---

## 6. テーブル名の命名規則
- テーブル名は **すべて大文字のスネークケース**とする。
- テーブル種別に応じて以下の接頭辞を付与する。
  - マスタテーブル：`M_` + 機能名
  - トランザクションテーブル：`T_` + 機能名
  - ビュー／中間テーブル：用途に応じて `V_`、`X_` を付与する

---

## 7. テーブルIDの扱い

- 本システムでは、**テーブルIDとして物理名をそのまま使用する**。
- テーブルIDと物理名は同一とし、別途管理用のIDは設けない。
- 設計書、DDL、ログ、問い合わせ対応においては、
  物理名を唯一の識別子として扱う。

---

## 8. 監査・排他カラムの標準（Audit / Versioning）

### 8.1 監査（全テーブル）
| 論理名   | 物理名       | 値の規約（抽象）                      |
|----------|--------------|---------------------------------------|
| 作成日時 | CREATED_AT   | 登録時刻（タイムゾーン一貫）          |
| 作成者   | CREATED_BY   | 登録主体（ユーザーID/システム識別子） |
| 更新日時 | UPDATED_AT   | 最終更新時刻                          |
| 更新者   | UPDATED_BY   | 最終更新主体                          |

- 物理名は **大文字スネークケース**。全テーブルで同一名を使用する。

### 8.2 排他（楽観ロック）
- 方式：**バージョン番号方式**（タイムスタンプ方式は採用しない）
- 物理名：**`VERSION`**
- ルール：0以上の整数。更新時に +1。更新前後で一致を検証。不一致は業務例外。

---

## 9. ログ出力および追跡におけるIDの扱い

### 9.1 基本方針
- 本サンプルでは、ログの一次的な追跡単位として `operationId` を用いる。
- `operationId` は 1 回の Application Service 呼び出しに対して 1 つ採番する内部相関 ID とする。
- 値形式は UUID とし、ログおよび内部コンテキストでのみ扱う。
- 業務的な追跡・問い合わせ対応では、以下を主要な識別子とする。
  - 画面ID（View/UI ID）
  - ユースケースID（UC-XXX）
  - アプリケーションサービスID（SASXXX）

### 9.2 内部サービスIDの扱い
- コマンドサービス（SCS）、クエリサービス（SQS）、ピュアサービス（SPS）にもIDを付与する。
- これらのIDは、ログ出力・デバッグ・設計書上の識別子として使用する。

### 9.3 実装上の制約
- 内部サービスID（SCS / SQS / SPS）は、
  クラス名・メソッド名・変数名には埋め込まない。
- 実装コードでは意味を表す名称を用いる。
- IDはログ項目・コメント・ドキュメントで参照する。
- `operationId` は利用者向け画面項目、公開 API 項目、業務データ項目として扱わない。
- `operationId` の生成・設定・破棄は framework 共通部で一元管理し、
  Controller、Application Service、Command Service 等の業務コードへ分散させない。

### 9.4 補足
- 内部サービスは設計上の都合により増減・分割・統合される可能性があるため、
  業務境界を表す識別子としては扱わない。
- 本規約に例外を設ける場合は、ADR に理由を記録する。
