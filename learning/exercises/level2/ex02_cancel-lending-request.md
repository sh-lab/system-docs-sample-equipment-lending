# 演習課題（Level2-02）
## 承認待ち貸出申請の取消対応

- 対象レベル：Level2  
- 主対象：Command Service 設計／状態更新処理  
- 変更種別：更新系機能追加（状態遷移あり・業務拡張なし）

---

## 背景

利用者から、次のような要望が出た。

> 「誤って貸出申請を行ってしまったため、  
> 管理者が承認する前であれば、申請を取り消したい。」

現在のシステムでは、貸出申請を行うと  
管理者による承認または却下が行われるまで、
利用者側から申請内容を取り消すことはできない。

---

## 課題内容

承認待ち状態の貸出申請について、  
**利用者が申請を取り消すことができる機能**を追加せよ。

本演習では、貸出申請の取消による

- 申請状態
- 備品状態

の更新を正しく行い、  
既存の業務ルールおよび設計方針との整合を保つことを求める。

---

## 機能要件（本演習スコープ）

- 利用者は、自身が行った貸出申請のうち  
  **承認待ち（PENDING_APPROVAL）状態の申請のみ**を取り消せること
- 承認済み、却下済み、返却申請中、返却完了の申請は取消不可とすること
- 申請を取り消した場合、対象申請は *取消済み* として扱われること
- 申請取消に伴い、対象備品の状態は貸出可能に戻ること

---

## 制約条件

- 既存の業務ルールおよび状態遷移の考え方を尊重すること
- Controller に業務判断や状態判定ロジックを実装してはならない
- Query Service を用いて更新処理を行ってはならない
- 更新処理は Command Service に集約すること
- 不整合なデータ状態を残してはならない
- 本変更は、貸出申請という業務エンティティに対する
  状態確定操作であるため、
  申請取消が確定した場合は
  貸出申請操作履歴（`H_LENDING_REQUEST_HISTORY`）に
  履歴を記録すること
- 当該履歴は業務利用者が参照することを想定せず、
  IT 担当による監査・追跡用途として扱うものとする

---

## 作業対象（ヒント）

- ユースケース設計
  - UC-002（備品の貸出を申請する）
  - UC-003（貸出申請を承認または却下する）
- 画面設計
  - 利用者貸出申請・返却画面（`V400`）
- 設計・実装
  - 貸出申請系 Command Service
  - 関連エンティティおよび Repository

---

## 状態遷移に関する補足

以下は一例であり、設計判断として整理してよい。

- 申請状態  
  - `PENDING_APPROVAL → CANCELED`  
    または  
  - `PENDING_APPROVAL → REJECTED`（却下と同義として扱う）

どの扱いを採用するかについては、  
**業務上の意味と設計上の整合性を踏まえて判断すること。**

---

## 求められる成果物

以下を **設計・実装の両面から** 整理せよ。

1. 必要な要件定義・ユースケース設計の修正または追記
2. Command Service の設計（新規作成または既存拡張の判断）
3. 設計に基づくソースコードの修正・追加
4. 状態遷移と更新処理の責務配置についての簡単な説明

---

## 考えるべきポイント

- 申請の「取消」と「却下」は業務的に同じか、異なるか
- 新しい Command Service を作るべきか、既存の却下処理を流用すべきか
- 申請取消時に、どのデータをどの順序で更新すべきか
- Controller が担うべき責務と、担ってはならない責務は何か
- 状態更新が途中で失敗した場合、どのように整合性を保つか

---

## 評価観点

本演習では、次の点を重視する。

- 状態遷移と業務意味が破綻していないか
- Command Service の責務が明確であるか
- 既存設計方針（責務分離・レイヤ構成）を遵守できているか
- 更新処理に一貫した判断理由があるか

唯一の正解は存在しないが、  
**設計思想と整合しない実装は不適切と判断される。**
