# ログ出力方針
（備品貸出管理システム／サンプルアプリ）

## 1. 目的
本書は、備品貸出管理システム（以下、本サンプル）における
ログ出力の基本方針を定義するものである。

本サンプルでは、業務フローおよび業務操作単位のトレーサビリティを
確保することを主目的とし、`info.log` に出力する対象を限定する。

---

## 2. 適用範囲
本方針は、本サンプルのアプリケーション実行時に出力する
アプリケーションログに適用する。

今回の対象は `info.log` とし、主に業務処理の開始・終了および
業務操作の実行履歴を記録する用途に用いる。

監視基盤、アクセスログ、デバッグログ、監査証跡の詳細設計は
本書の適用対象外とする。

---

## 3. ログ出力の目的
本サンプルにおけるログ出力の目的は以下のとおりである。

- 業務フローの開始および終了を追跡できるようにすること
- 管理者・利用者による業務操作の実行単位を追跡できるようにすること
- 障害調査時に、どのユースケースがどの順序で実行されたかを確認できるようにすること

一方で、本サンプルでは過剰なログ出力を避け、
可読性および運用時の確認容易性を優先する。

---

## 4. 基本方針
- `info.log` には、**Application Service の開始・終了** と
  **Command Service の実行** のみを出力する。
- Controller、Query Service、Repository では、
  業務トレース目的の `INFO` ログを出力しない。
- 業務コードへ個別のログ出力処理を散在させず、
  横断的関心事として一貫した方式で扱うことを前提とする。
- ログは、障害解析に必要な最小限の情報に限定し、
  業務上の自由記述コメントや不要な詳細データを原則として出力しない。
- 個人情報や機微情報を含む可能性がある値は、
  原則として `info.log` にそのまま出力しない。

---

## 5. `info.log` の出力対象

### 5.1 出力対象
`info.log` の出力対象は、以下に限定する。

| 対象レイヤ | 出力契機 | 目的 |
|------------|----------|------|
| Application Service | 開始 | ユースケース開始の記録 |
| Application Service | 終了 | ユースケース終了の記録 |
| Command Service | 実行 | 業務更新操作の実行記録 |

### 5.2 非対象
以下は `info.log` の出力対象外とする。

- Controller のリクエスト受信・画面遷移
- Query Service の検索・参照処理
- Repository の SQL 実行や永続化詳細
- 認証・認可処理の詳細トレース
- 開発・検証目的のデバッグログ
- 例外スタックトレースを伴うエラーログの詳細方針

---

## 6. 出力内容の方針
`info.log` に出力する内容は、業務トレースに必要な最小限の識別情報とする。

想定する出力項目は以下のとおりである。

- 実行日時
- ログレベル
- 対象クラスまたはサービス名
- イベント種別（開始、終了、実行）
- `operationId`
- 業務処理を識別できる最小限の識別子

出力内容の詳細ルールは以下とする。

- 自由記述コメント本文は出力しない。
- 業務データの全文や一覧件数の詳細は原則として出力しない。
- 必要に応じて申請ID、利用者ID等の識別子を出力対象とするが、
  個人情報や機微情報の直接出力は避ける。
- `operationId` は 1 回の Application Service 呼び出し単位で採番し、
  同一業務操作に含まれる Application Service 開始・終了および
  Command Service 実行ログで共通利用する。

---

## 7. 実現方式の考え方
- テンプレート方針に従い、ログ出力は横断的に適用できる仕組みで実現する。
- 本サンプルでは、
  Spring AOP により Application Service と
  Command Service に対して限定的に `INFO` ログを付与する方式を採用する。
- Spring AOP の Pointcut では、Application Service の開始・終了と
  Command Service の実行のみを捕捉対象とし、
  `info.log` の出力範囲を本方針に従って制御する。
- Application Service の開始時に `operationId` を生成し、
  同一スレッド内の処理で参照できるコンテキストへ設定する。
- Application Service の終了時は、成功・失敗を問わずコンテキストを必ず破棄し、
  次の業務操作へ `operationId` を持ち越さない。
- 業務コード内へ都度 `logger.info(...)` を記述する方式は、
  方針不統一および保守性低下を招くため採用しない。

---

## 8. 他方針との関係
- 例外・障害時のログ出力方針は `docs/02_architecture/error-handling.md` と整合させる。
- トランザクション境界は `docs/02_architecture/transaction-boundaries.md` に従い、
  `Application Service` を単位として業務フローを記録する。
- 本方針に例外を設ける場合は、その理由を設計判断として明確化する。

---

## 9. 補足
- 本書はログ出力対象と粒度の方針を定めるものであり、
  実際のログ設定ファイル、ローテーション方針、保存期間、出力フォーマット詳細は
  実装時に別途具体化する。
- ログ出力方式の具体実装は、Spring Framework の標準機能である
  Spring AOP を前提として設計する。
- 本サンプルでは、`src/main/java/net/shlab/hogefugapiyo/framework/logging` 配下に
  Spring AOP によるログ出力機構を配置し、`src/main/resources/logback-spring.xml` に
  `info.log` の出力設定を定義する。
- `operationId` の生成・保持は `src/main/java/net/shlab/hogefugapiyo/framework/core` 配下の
  共通コンテキスト機構で扱い、業務コードから直接 `ThreadLocal` を操作しない。
