# ADR-000 system-docs-template の採用

## ステータス
採用

## コンテキスト
本プロジェクトは、業務システム開発における設計・実装の考え方を
学習用途として示すサンプルアプリケーションである。

単に動作するコードを示すのではなく、
要求整理から設計、実装への落とし込みまでを
一貫した形で理解できることを重視している。

そのためには、設計ドキュメントの構造と粒度を
あらかじめ明示した上で作業を進める必要がある。

## 決定
本サンプルアプリケーションでは、
設計ドキュメント構成の前提として
**system-docs-template** を採用する。

要件定義、アーキテクチャ設計、詳細設計、実装支援といった
各フェーズのドキュメントは、
system-docs-template が定義する構成・責務分離に従って作成する。

## 理由
- 業務システム開発で一般的に求められるドキュメント構造を、
  明示的なテンプレートとして提供できるため。
- 設計レイヤごとの責務（Why / What / How）を分離しやすく、
  学習用途として理解しやすいため。
- GitHub Copilot を用いた設計・実装補助において、
  前提構造を固定することで出力のばらつきを抑制できるため。
- サンプルアプリとテンプレートを分離することで、
  「考え方」と「具体例」の対応関係を明確に示せるため。

## 代替案
- ドキュメント構成を定義せず、自由形式で設計を進める。
- 既存フレームワークや書籍の構成にそのまま従う。

いずれも、学習者にとって
「なぜその構成になっているのか」を理解しにくくなるため採用しない。

## 結果・影響
- 本サンプルのすべての設計・実装判断は、
  system-docs-template を前提として行われる。
- 実装支援情報は、設計ドキュメントと分離し、
  `04_implementation` 配下に配置する。
- system-docs-template を採用していないプロジェクトへ
  そのまま適用することは想定しない。
