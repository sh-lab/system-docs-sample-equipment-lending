# 演習課題（Level1-02）
## 備品名の OR 検索対応

- 対象レベル：Level1
- 主対象：Query Service／Query Repository
- 変更種別：既存検索仕様の拡張（参照系・更新なし）

---

## 背景

利用者から、次のような要望が出た。

> 「複数の備品候補を一度に探したい。  
> たとえば『長机』と『プロジェクター』のどちらかが
> 含まれる備品をまとめて検索したい。」

現在の備品検索では、  
備品名は単一キーワードによる部分一致検索のみを提供している。

---

## 課題内容

備品検索画面において、  
備品名に **複数のキーワード** を指定した場合に、

- いずれかのキーワードを含む備品を検索できるようにせよ  
  （OR 検索）。

既存の検索条件（備品種別、貸出状態 等）は変更しないものとする。

---

## 機能要件（本演習スコープ）

- 利用者は、備品名欄に複数のキーワードを指定できること
- 指定されたキーワードの **いずれか** を含む備品を検索できること
- 検索結果の表示方式や件数制限は現行仕様を踏襲すること

※ AND 検索への変更は行わない。

---

## 制約条件

- Controller に業務判断や検索条件解釈ロジックを追加してはならない
- 既存のアーキテクチャ方針（責務分離）を逸脱してはならない
- 検索仕様の変更は、参照系の設計として完結させること
- 新たな状態遷移や更新処理は追加しないこと

---

## 作業対象（ヒント）

- 機能要件
  - FR-001 備品検索
- 画面設計
  - 備品検索画面（`V300`）
- 設計・実装
  - 備品検索系 Query Service
  - 備品検索系 Query Repository

---

## 求められる成果物

以下を **設計・実装の両面から** 整理せよ。

1. 必要に応じた要件定義・画面設計の修正
2. Query Service / Query Repository における検索仕様の変更
3. 設計に基づくソースコードの修正
4. 検索条件解釈の責務をどの層に置いたかの簡単な説明

---

## 考えるべきポイント

- 備品名入力値を **どの層で解釈するのが妥当か**
- OR 検索という仕様は、業務判断か、検索判断か
- Controller で処理したくなる誘惑をどう回避するか
- 既存検索条件との責務分離は保たれているか

---

## 評価観点

本演習では、次の点を重視する。

- Query Service / Repository の責務が明確であるか
- Controller に業務的な判断が入り込んでいないか
- 既存の設計方針を崩さずに仕様変更できているか
- 設計判断に一貫した理由があるか

唯一の正解は存在しないが、  
**責務分離を崩す実装は不適切と判断される。**
