# ADR-006 重複送信防止の多層防御

## ステータス
採用

## コンテキスト
本サンプルアプリケーションでは、更新系画面の成功後に
Post/Redirect/Get（PRG）を適用している。

PRG は、更新完了後の画面再読み込みによる再送防止には有効である。
一方で、送信前の連打、通信待ち中の複数回押下、同一フォームの多重 POST までは防げない。

また、`V400`、`V500`、`V700` の画面設計書では、
更新系ボタン押下後の二重送信防止を前提としているが、
実装はその意図を十分に表現できていなかった。

本サンプルは教育用途であり、
業務 Web アプリケーションにおける
クライアント側とサーバ側の役割分担を説明しやすい構成が求められる。

## 決定
本サンプルアプリケーションでは、更新系フォームの重複送信防止として
**PRG + 一時非活性化 + ワンタイムトークン** を併用する。

- **PRG**
  - 成功後の再読み込みによる再送を防止する。
- **一時非活性化**
  - クライアント側で送信直後の連打を抑止する。
- **ワンタイムトークン**
  - サーバ側で同一フォームの重複受理を防止する。

対象は更新系 POST フォームを基本とし、
本サンプルでは `V400`、`V500`、`V700` を主対象とする。

## 理由
- PRG のみでは、最初の POST が複数回送信される事象を防げないため。
- 一時非活性化のみでは、JavaScript 無効化や多重リクエスト時の最終防衛線にならないため。
- ワンタイムトークンのみでは、利用者への操作フィードバックが弱く、
  教育用サンプルとしてクライアント側の責務を示しにくいため。
- 多層防御とすることで、
  「再送防止」「連打抑止」「重複受理防止」の違いを学習しやすくなるため。
- 共通部へ集約することで、
  各 Controller やテンプレートに個別実装を散在させずに済むため。
  サーバ側のワンタイムトークン検証は `HandlerInterceptor` に集約する。

## 代替案
- PRG のみを採用する。
- PRG + 一時非活性化のみを採用する。
- PRG + ワンタイムトークンのみを採用し、一時非活性化を省略する。
- 楽観ロックや業務例外のみで重複更新に対応する。

これらはいずれも一部には合理性があるが、
本サンプルでは教育用途としての説明容易性と
実務的な防御の分かりやすさを両立するため採用しない。

## 結果・影響
- 更新系フォームでは、CSRF トークンに加えて
  重複送信防止用のワンタイムトークンを扱う。
- ワンタイムトークンはセッション上の一時状態として管理するが、
  認証・認可の正としては扱わない。
- ワンタイムトークンの検証・消費は更新系 POST の前段で `HandlerInterceptor` が実施し、
  Controller は表示時のトークン払い出しと通常の画面責務に限定する。
- ワンタイムトークンが不正または再利用済みである場合は、元画面へ戻さず、多重送信検知専用画面へ遷移させる。
- クライアント側では、対象フォーム送信時に
  送信ボタンを一時非活性化する共通スクリプトを適用する。
- Web スライステストでは、PRG と CSRF に加え、
  ワンタイムトークンの正常系・不正系を確認対象に含める。
- CSRF トークンとワンタイムトークンは目的が異なるため、
  設計書および実装上で混同しない。
