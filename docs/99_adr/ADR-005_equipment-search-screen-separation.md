# ADR-005 利用者向け備品検索と管理者向け備品検索の分離

## ステータス
採用

## コンテキスト
本サンプルでは、
利用者向けの備品検索機能に加えて、
管理者が備品を検索し、
新規備品登録および備品情報更新へ遷移するための
`管理者備品検索` 画面を追加する。

このとき、
利用者向け備品検索と管理者向け備品検索を
同一画面または同一要件として扱うか、
用途ごとに分離するかを判断する必要がある。

利用者向け備品検索は、
貸出対象となる備品を見つけることが主目的であり、
貸出可能かどうかを中心に確認できればよい。

一方で管理者向け備品検索は、
備品台帳の維持管理を目的とし、
備品登録、備品情報更新、登録済み備品の確認へ
遷移するための起点画面となる。

両者には、少なくとも以下の差異がある。

- 検索条件が異なる
  - 利用者向けは貸出対象の絞り込みが中心である。
  - 管理者向けは備品状態や `システム登録日` を含む管理項目での絞り込みが必要である。
- 検索結果に求められる項目が異なる
  - 利用者向けは貸出可否を判断するための表示が中心である。
  - 管理者向けは備品状態や `システム登録日` を確認し、
    管理操作へ進むための表示が必要である。
- 今後の機能追加の方向が異なる
  - 利用者向けは予約や利用申請補助など、
    利用業務に寄った拡張が想定される。
  - 管理者向けは棚卸し、利用停止判断、台帳確認など、
    管理業務に寄った拡張が想定される。

学習用サンプルとしても、
異なる責務を一つの画面や一つの要件に混在させると、
画面の目的と将来の拡張方針が読み取りにくくなる。

## 決定
本サンプルでは、
利用者向け備品検索と管理者向け備品検索を
別画面・別要件として分離する。

要件定義では、
利用者向け検索は従来どおり `備品検索` として扱い、
管理者向け検索は `管理者備品検索` として独立した機能要件、
ユースケース、および用語定義を持たせる。

## 理由
- 利用者と管理者では、
  検索の目的が異なり、
  求められる検索条件が一致しないため。
- 利用者と管理者では、
  検索結果に表示すべき項目が異なり、
  同一画面に集約すると責務が曖昧になりやすいため。
- 管理者向け検索は、
  備品登録や状態更新へ遷移する業務起点であり、
  利用者向け検索より強い管理責務を持つため。
- 将来の機能追加において、
  利用者向けと管理者向けで拡張方向がさらに分かれることが見込まれるため。
- 学習用サンプルとして、
  役割ごとの責務差分を要件・画面単位で明確にした方が
  設計意図を理解しやすいため。

## 代替案
- 利用者向けと管理者向けで同一の備品検索画面を共用する。
- 単一の備品検索要件の中で、
  権限に応じて検索条件や表示項目だけを切り替える。

これらの案は実装上は可能であるが、
要件定義上の責務境界が曖昧になりやすく、
将来の管理機能追加時に
利用者向け要件と管理者向け要件が相互に影響しやすい。

また、画面目的の説明が複雑になり、
学習用サンプルとしての見通しも損ないやすいため、
本サンプルでは採用しない。

## 結果・影響
- `docs/01_requirements/functional-requirements.md` では、
  利用者向けの `備品検索` と
  管理者向けの `管理者備品検索` を別機能として定義する前提となる。
- `docs/01_requirements/use-cases/` では、
  管理者向け検索を独立したユースケースとして定義する。
- `docs/01_requirements/glossary.md` では、
  `管理者備品検索` と `システム登録日` を
  管理業務上の用語として定義する。
- 後続の設計では、
  利用者向け検索と管理者向け検索を
  画面、検索条件、検索結果項目、遷移先の観点で
  別責務として整理しやすくなる。
- 今後の機能追加時にも、
  利用者向け検索と管理者向け検索を
  それぞれ独立して拡張しやすくなる。
