# 演習課題（Level2-01）
## 管理者による備品別貸出履歴の参照

- 対象レベル：Level2  
- 主対象：Query Service 設計／画面追加／参照系責務分離  
- 変更種別：機能追加（参照系・更新なし）

---

## 背景

管理者から、次のような要望が出た。

> 「備品ごとの現在の状態だけでなく、  
> その備品がこれまでにどのように貸し出されてきたかを確認したい。」

現在のシステムでは、管理者は  
管理者備品検索画面（`V600`）から備品の検索や登録・更新を行うことはできるが、  
**特定の備品に対する過去の貸出履歴をまとめて確認する手段は提供されていない。**

---

## 課題内容

管理者備品検索画面（`V600`）から単一の備品を選択し、  
当該備品の **これまでの貸出情報を時系列で一覧表示する画面** を追加せよ。

本演習では、**貸出履歴の参照のみ** を扱い、  
状態更新や業務判断の追加は行わないものとする。

---

## 機能要件（本演習スコープ）

- 管理者は、管理者備品検索画面から単一の備品を選択できること
- 選択した備品に対し、別画面へ遷移できること
- 遷移先画面では、当該備品に紐づく過去の貸出情報を時系列で一覧表示できること
- 一覧表示は参照専用とし、更新操作は含めないこと

---

## 表示対象となる情報（例）

以下は一例であり、設計判断により調整してよい。

- 貸出申請ID
- 利用者
- 貸出開始日時
- 返却完了日時
- 申請結果（承認／却下／返却完了 など）

※ 集計（回数、期間合算など）は行わない。

---

## 制約条件

- 既存の業務ルールおよび状態遷移定義は変更しないこと
- 新たな状態遷移、更新処理、Command Service は追加しないこと
- Controller に業務判断やデータ取得ロジックを実装してはならないこと
- 既存のアーキテクチャ方針（責務分離）を遵守すること

---

## 作業対象（ヒント）

- 画面設計  
  - 管理者備品検索画面（`V600`）  
  - 新規に追加する履歴参照画面
- Query Service / Query Repository
- 貸出申請・申請明細・履歴関連テーブル

※ どのテーブルや履歴情報を正として参照するかは、  
設計判断として整理すること。

---

## 求められる成果物

以下を **設計・実装の両面から** 整理せよ。

1. 必要な画面遷移・画面設計の修正または追加
2. Query Service および Query Repository の設計
3. 設計に基づくソースコードの修正・追加
4. 本機能をどの責務として切り出したかの簡単な説明

---

## 考えるべきポイント

- 「備品の利用履歴」とは、どの情報を指すのか
- どのデータを正として貸出履歴を構成するのが妥当か
- 既存 Query の拡張と、新規 Query Service 追加のどちらが適切か
- Application Service を介在させる必要はあるか
- 画面と Query の責務分離は保たれているか

---

## 評価観点

本演習では、次の点を重視する。

- 参照系として適切なレイヤ構成になっているか
- Query Service の責務が明確であるか
- UI 都合のロジックを Service や Repository に持ち込んでいないか
- 設計判断に一貫した理由があるか

唯一の正解は存在しないが、  
**設計方針と整合しない実装は不適切と判断される。**
